# 客户级产品化交付规划

## 目标

把 Codex 历史管理器从“本机可用工具”推进为“客户电脑可直接运行的软件产品”。软件应能在不同 Windows 电脑、不同安装位置、不同 Codex 数据目录下稳定启动，支持清晰的首次使用引导、诊断信息、导出备份和错误恢复。

## 当前核心风险

1. 可写目录依赖程序所在目录。若 EXE 位于 `Program Files`、只读目录、网盘同步目录或 PyInstaller 临时解包目录，导出、备份、日志可能写入失败。
2. Codex 数据目录未持久化。用户手动选择数据目录后，下次启动仍可能回到默认目录。
3. 首次启动缺少产品化引导。客户电脑没有 `.codex`、目录不完整或 Codex 版本变化时，界面需要说明原因和下一步操作。
4. 诊断信息还不够完整。售后排查需要同时知道应用目录、资源目录、用户数据目录、配置文件、日志和 Codex 数据目录状态。
5. 发布包需要按“普通客户下载 EXE 后直接使用”的思路设计，不能依赖开发者机器路径。

## 设计原则

1. 程序目录只放资源和可执行文件，不作为默认写入位置。
2. 用户数据统一写入 `%LOCALAPPDATA%\CodexHistoryManager`，支持 `CODEX_HISTORY_MANAGER_HOME` 环境变量覆盖。
3. 配置文件保存到用户数据目录，记录最近一次有效 Codex 数据目录。
4. 没有数据时不报错、不空白，显示明确的空状态和操作建议。
5. 诊断中心能一键复制关键状态，便于客户反馈问题。

## 本阶段开发项

### 1. 用户数据目录

新增应用数据根目录：

- Windows 默认：`%LOCALAPPDATA%\CodexHistoryManager`
- 其他平台兜底：`~/.codex-history-manager`
- 环境变量覆盖：`CODEX_HISTORY_MANAGER_HOME`

子目录：

- `exports`：导出文件和压缩包
- `backups`：会话备份
- `logs`：应用日志和操作日志
- `config.json`：用户配置

### 2. 配置持久化

配置内容：

```json
{
  "codex_root": "C:/Users/<User>/.codex",
  "updated_at": "2026-06-06T00:00:00Z"
}
```

启动逻辑：

1. 若配置中的 `codex_root` 存在，优先使用。
2. 否则使用 `CODEX_HOME`。
3. 否则使用 `~/.codex`。
4. 用户在界面选择数据目录后立即保存配置。

### 3. 首次启动引导

当未发现 `state_5.sqlite`、`session_index.jsonl` 或 `sessions` 目录时：

- 会话列表显示“尚未发现 Codex 会话数据”。
- 预览区显示原因和建议：
  - 打开 Codex 并产生至少一次对话；
  - 点击“选择数据目录”选择正确的 `.codex` 目录；
  - 在诊断中心复制信息用于排查。

### 4. 诊断中心增强

诊断信息新增：

- 用户数据目录
- 配置文件路径
- 配置是否存在
- 当前 Codex 数据目录来源
- 导出、备份、日志目录可写性

### 5. 测试与发布验证

必须通过：

- 公开仓库自检，使用临时 `.codex` 夹具，不读取真实用户数据。
- 冒烟测试，验证真实本机 Codex 数据读取、导出、备份。
- 语法解析检查。
- `git diff --check`。
- 敏感信息扫描。
- GitHub Actions CI 通过。

## 后续产品化路线

1. 增加 NSIS 或 Inno Setup 安装包，支持开始菜单、桌面快捷方式和卸载。
2. 增加自动检查更新或“打开最新版发布页”。
3. 增加客户反馈包导出功能，把诊断、日志摘要、版本号打包，不包含聊天正文。
4. 增加多语言扩展能力，但默认保持中文。
5. 增加更完整的用户手册和故障排查手册。
