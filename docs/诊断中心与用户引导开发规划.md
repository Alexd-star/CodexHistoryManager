# 诊断中心与用户引导开发规划

## 一、目标

本轮目标是补齐普通用户使用时最容易卡住的部分：出错后不知道原因、不知道日志在哪里、不知道该复制什么信息给开发者、不知道备份/导出/日志目录在哪里。

开发完成后，软件应具备：

- 可复制的诊断信息；
- 最近操作日志查看；
- 最近备份记录查看；
- 一键打开导出、备份、日志目录；
- 错误发生时写入诊断日志；
- 错误弹窗包含“下一步建议”，而不是只显示 Python 异常文本。

## 二、最佳实践依据

### Python logging

Python 官方 logging HOWTO 建议使用 `logging` 记录运行时事件。对于异常场景，使用 `logger.exception(...)` 可以自动记录堆栈，便于定位根因。

落地方式：

- 在 `app.py` 中创建应用日志文件 `logs/应用日志.log`；
- 普通操作继续写 `logs/操作日志.jsonl`；
- 异常和诊断写入 `logs/应用日志.log`；
- 日志目录继续被 `.gitignore` 排除，避免公开聊天路径和本地环境。

### tkinter messagebox

Python tkinter messagebox 适合展示 `showinfo`、`showerror` 等用户提示。它不适合承载大量诊断文本，因此错误弹窗只放简短摘要，完整诊断放到诊断中心和日志文件。

### SQLite 与备份

当前项目已经使用 SQLite backup API 备份状态数据库，这是正确方向。下一步不是重写备份，而是让用户能看到最近备份记录和快速打开备份目录。

## 三、用户体验设计

### 诊断中心放置位置

放在桌面版“管理”页下方，原因：

- 管理页已经承载备份、恢复、Web 服务；
- 诊断、备份、日志都属于维护类操作；
- 不打扰主要的预览和导出流程。

### 诊断中心内容

诊断中心显示：

- 应用版本；
- Python 版本；
- 运行模式：源码/EXE；
- 应用目录；
- Codex 数据目录；
- SQLite 状态数据库是否存在；
- 会话索引是否存在；
- 会话目录是否存在；
- 导出目录；
- 备份目录；
- 日志目录；
- 当前会话数量；
- 已选会话数量；
- 最近操作记录；
- 最近备份记录。

### 用户操作

提供按钮：

- 刷新诊断；
- 复制诊断信息；
- 打开导出目录；
- 打开备份目录；
- 打开日志目录。

## 四、工程实现范围

### 修改 `app.py`

新增：

- `APP_VERSION`
- `APP_LOG`
- `setup_logging()`
- `log_exception(context, exc)`
- `CodexStore.diagnostic_snapshot()`

目标：

- 后端可以生成结构化诊断信息；
- 任何桌面端和 Web 端都能复用。

### 修改 `modern_app.py`

新增：

- 管理页诊断中心；
- 诊断文本框；
- 复制诊断；
- 打开导出/备份/日志目录；
- 错误处理增强：写日志，弹窗给出简短建议。

### 修改文档

- README 增加诊断中心说明；
- 设计说明增加日志和诊断策略；
- CHANGELOG 增加未发布内容。

## 五、验收标准

- `python .\tests\公开仓库自检.py` 通过；
- `python .\tests\冒烟测试.py` 通过；
- `python .\modern_app.py` 可打开窗口；
- 管理页能显示诊断信息；
- 复制诊断信息不报错；
- 打开导出/备份/日志目录不报错；
- 人为触发错误时 `logs/应用日志.log` 中有异常记录；
- GitHub Actions CI 通过。
